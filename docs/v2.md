# Prospecting Bot — V2 Features

## 1. Batch Account Input

**What:** Rep can drop a list of 5–20 accounts and the bot processes them in parallel — one workflow per account, no manual babysitting per session.

**Why:** AJ's operating model is "give it a list, let it run while I focus on positioning." The current one-session-at-a-time architecture with manual approval checkpoints doesn't scale to how he actually works.

**Scope:**
- Accept a newline- or comma-separated list of accounts in a single message
- Spawn a session per account, run research and contact sourcing concurrently
- Deliver a consolidated summary card in Slack when all accounts are done (or per-account as each completes)
- Rep reviews and approves personas/sequences per account at their own pace — checkpoints stay, but don't block other accounts from processing

**Open questions:**
- Approval UX when 10 accounts finish simultaneously — thread per account vs. single digest
- Token/API cost implications of concurrent research runs

---

## 2. Follow-up Automation (LinkedIn + DMs)

**What:** After sequences are approved, the bot automates the first outreach touches — LinkedIn connection requests and/or DMs — rather than delivering a brief that the rep has to manually execute.

**Why:** AJ explicitly called out LinkedIn requests and DMs as part of the leg work he wants automated. Right now the bot stops at sequence delivery.

**Scope:**
- LinkedIn connection request automation (triggered post-approval, per persona)
- LinkedIn DM send for 1st-degree connections
- Follow-up scheduling — if no response in N days, queue next step
- Tracking back to session/persona record in Postgres so rep can see status

**Dependencies:**
- LinkedIn automation layer (Phantombuster, Dux-Soup, or direct LinkedIn API if available)
- Follow-up scheduler (cron or Railway background worker)
- Rep opt-in per sequence — automation should be explicit, not default

**Open questions:**
- LinkedIn ToS risk tolerance — scraping-based tools carry account risk
- Whether follow-up scheduling lives in the bot or routes through an existing SEP (SendGrid sequences already in stack)
